You are a senior backend architect and database performance expert.
You must design and implement a production-grade API written in Go (Golang).

GOAL:
Build a high-performance Go API that serves:
- A web GUI
- A Telegram bot
- A WhatsApp bot

The API sits in front of a very large PostgreSQL database (2+ TB) and must remain fast, safe, and scalable.
The API must automatically reflect newly inserted PostgreSQL data without code changes.

IMPLEMENTATION LANGUAGE (MANDATORY):
- Go (Golang) ONLY
- Use idiomatic Go practices
- Use standard net/http or a lightweight router
- Use pgx for PostgreSQL access
- Redis for caching
- PgBouncer-compatible connection handling

==============================
MANDATORY ARCHITECTURE RULES
==============================

1. DATABASE & CONNECTIONS
- PostgreSQL is the single source of truth.
- The API must not preload or cache full tables in memory.
- New rows inserted into PostgreSQL must appear automatically in API responses.
- Use prepared statements only.
- All database access must go through a shared data access layer.
- Design for PgBouncer (transaction pooling).

2. QUERY RULES (ABSOLUTE)
- NEVER use SELECT *
- NEVER use OFFSET pagination
- ALWAYS use LIMIT (hard max: 50)
- ALWAYS use keyset (cursor-based) pagination
- Queries must use indexed, immutable columns (id or created_at).
- Stream rows using pgx.Rows (no full result loading).

3. ADAPTIVE DATA ACCESS
- API endpoints must not assume dataset size.
- Sorting must be restricted to whitelisted indexed columns.
- Filters must be whitelisted and index-backed.
- Table metadata may be read at runtime if needed, but raw SQL must never be exposed.

4. CACHING (REQUIRED)
- Implement Redis caching with a read-first strategy:
    IF cache hit → return immediately
    ELSE → query DB → cache → return
- Cache keys must be derived from:
    endpoint + normalized filters + cursor + limit
- Cache TTL must be short (10–60 seconds) to ensure visibility of new data.

5. API DESIGN
- RESTful or JSON-based API.
- Fixed endpoints with strict input validation.
- Hard limits on:
    - page size
    - filters
    - sort fields
- Responses must be shaped specifically for clients (GUI and bots).

6. PAGINATION ALGORITHM
- Cursor value comes from the last row of the previous response.
- Pagination performance must remain constant regardless of table size.
- Cursor must be opaque to the client.

7. BOT SUPPORT
- The API must support webhook-based clients (Telegram and WhatsApp).
- Bot commands must map to safe, predefined API queries.
- Bots must never trigger unbounded or expensive queries.
- Shared business logic must be reused between web and bot clients.

8. PERFORMANCE & SCALABILITY
- Assume tables can exceed hundreds of millions of rows.
- Support partitioned tables.
- Support composite indexes matching WHERE + ORDER BY.
- Designed to scale horizontally.
- Read replicas may be added without code changes.

9. SECURITY & STABILITY
- No raw SQL input from users.
- Strict parameter validation.
- Rate limiting per client.
- Context-based timeouts on all DB calls.
- Graceful error handling.

==============================
OUTPUT REQUIREMENTS
==============================
Provide:
1. API architecture overview
2. Core request-handling algorithm
3. Go project structure
4. Example Go HTTP handler
5. Example pgx query with keyset pagination
6. Redis caching example in Go
7. Example webhook handler for a bot
8. Example composite index SQL

==============================
ABSOLUTE PROHIBITIONS
==============================
- No OFFSET
- No SELECT *
- No dynamic SQL
- No frontend DB access
- No unbounded queries
- No loading large datasets into memory

FINAL OBJECTIVE:
The resulting Go API must:
- Remain fast at 2+ TB scale
- Automatically surface newly inserted data
- Serve web and bot clients safely
- Enforce database best practices by design
